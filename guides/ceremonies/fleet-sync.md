# Fortnightly — Fleet Sync

**Duration:** 30 minutes
**Facilitator:** Engineering leadership + Product leadership
**When:** Every two weeks, only if you run multiple boats
**Attendees:** One representative per boat, usually the Captain

---

## Purpose

Surface cross-boat dependencies before they become blockers. Align on shared platform priorities. Make risks affecting multiple boats visible early.

Fleet Sync is not a status meeting and not a second Route Planning session. Boat-local shaping stays inside each boat.

---

## Before the Meeting (Facilitator, 10 min prep)

- [ ] Ask each boat for any cross-boat dependency, shared platform request, or multi-boat risk
- [ ] Group similar platform asks together
- [ ] Carry forward any unresolved items from the previous Fleet Sync
- [ ] Remove boat-local issues that should stay in Route Planning

---

## Script

| Time | Activity | Detail |
|---|---|---|
| 0–5 min | **Review open items** | What carried over from the last Fleet Sync? Anything resolved, blocked, or now irrelevant? |
| 5–15 min | **Dependencies** | Which boats are waiting on which other boats? Name the owner, next step, and timing risk. |
| 15–25 min | **Shared platform priorities** | Are multiple boats asking for the same platform capability? Decide priority, sequence, or deferral. |
| 25–30 min | **Cross-boat risks** | Flag risks that affect multiple boats. Name owner, mitigation, and whether leadership action is needed. |

---

## Outputs

- [ ] Each cross-boat dependency has a requesting boat, owning boat, next step, and target date
- [ ] Shared platform asks are explicitly prioritised or deferred
- [ ] Multi-boat risks are visible with an owner and mitigation
- [ ] Any needed leadership escalations are named immediately

---

## Key Questions to Ask

- "What is likely to block another boat in the next two weeks?"
- "Which dependencies need a named next step today?"
- "Are multiple boats asking for the same platform work?"
- "What risk would surprise more than one boat if we ignored it?"
- "Does this need leadership trade-off, or can the boats resolve it directly?"

---

## Anti-Patterns

| ❌ Don't | ✅ Do |
|---|---|
| Turn it into round-robin status updates | Only discuss items crossing boat boundaries |
| Re-plan a boat's internal work | Keep boat-local shaping in Route Planning |
| Let dependencies stay vague | Name the requesting boat, owning boat, next step, and target date |
| Treat every platform request as urgent | Make explicit priority calls and defer openly when needed |
| Leave risks implied | Name the risk, owner, and mitigation |
